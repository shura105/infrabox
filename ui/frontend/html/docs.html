Огляд системи

Infrabox — домашня SCADA-система для моніторингу та управління інженерними підсистемами об'єкта. Дані збираються через MQTT, зберігаються в Redis і відображаються на налаштовуваних екранах у реальному часі через WebSocket.

Основні можливості

  • Відображення значень параметрів у реальному часі (WebSocket)
  • Налаштовувані SCADA-екрани з довільним розташуванням елементів
  • Три типи екранів: PC, Render (пряма передача зображення), Mobile
  • Підтримка кількох фізичних пристроїв відображення на один тип
  • Кольорова індикація якості сигналу (GOOD / WARN / ALARM…)
  • Архівування всіх параметрів із побудовою трендів
  • Симулятор для тестування без реального обладнання
  • Рольова система доступу з правами на рівні сторінок та об'єктів

Архітектура

Система розгорнута у Docker-контейнерах і складається з кількох незалежних сервісів.

:80 → :443 · HTTPS-точка входу

Роздає статику UI, проксіює API та WebSocket. SSL termination.

⚙ infrabox-backend
:8090 · FastAPI + uvicorn

REST API для екранів та проекту. WebSocket-стрім даних. Керування симулятором.

🔑 infrabox-auth
:8095 · FastAPI

Авторизація, JWT-токени. Зберігає користувачів та права в Redis.

↕

:6379

🦟 infrabox-mosquitto-real
:1883 · MQTT broker

Приймає дані від реального обладнання.

🤖 infrabox-simulator

Генерує тестові дані замість реального обладнання.

↕

Безперервний запис всіх параметрів. Побудова трендів.

Перегляд екранів

Головна сторінка (/) відображає SCADA-екрани у реальному часі. При завантаженні автоматично відкривається перший екран, що відповідає обраному пристрою відображення.

Навігація

  • Кнопки Навігація та картки Підсистема переходять на інший екран того ж типу та пристрою
  • Кнопка ← в хедері повертає на попередній екран (стек навігації)
  • Назва поточного екрану відображається по центру хедера
  • Переходи між екранами різних типів або різних пристроїв заблоковані

Статус з'єднання

Нижній рядок показує стан WebSocket:

  • ● Live — дані надходять у реальному часі
  • ● — підключення встановлюється
  • ● — з'єднання перервано, виконується повторне підключення з exponential backoff

Пристрої відображення

Кожен екран у системі прив'язаний до конкретного пристрою відображення — числового параметра D1, D2, D3… Це дозволяє мати окремі набори екранів для різних фізичних дисплеїв одного типу.

Вибір пристрою

У меню ☰ Меню → Пристрій відображення відображається список пристроїв, доступних для поточного типу екрана. Вибір зберігається в localStorage.

Фільтрація: меню показує лише ті номери пристроїв, для яких є екрани того ж типу що й поточний (PC/Render/Mobile). Якщо для PC немає екранів D2 — D2 в меню не з'явиться.

Початковий екран

При завантаженні або зміні пристрою система завантажує екран з найменшим screenNum серед екранів обраного пристрою.

Конструктор — огляд

Сторінка /constructor — лендінг з вибором напрямку конструювання:

  • Структура і екрани (/constructor-screens) — побудова структури екранів та редагування їх елементів
  • Обладнання (/constructor-equipment) — конфігурація обладнання (в розробці)

Доступний тільки авторизованим адміністраторам.

Режими роботи конструктора екранів

  • Структура — управління переліком екранів: створення, видалення, розташування на канвасі або в таблиці
  • Елементи — редагування вмісту конкретного екрана (подвійний клік на екрані в режимі структури)
Зберігання: зміни зберігаються кнопкою 💾 Зберегти. Автозбереження відсутнє.

Типи та нумерація екранів

Кожен екран належить до одного з трьох типів:

P — PC R — Render M — Mobile
ТипПрефіксПризначення
PCPВеб-інтерфейс для комп'ютерних браузерів
RenderRПряма передача зображення на дисплей (bitmap по мережі, тільки дельти)
MobileMКомпактний інтерфейс для мобільних пристроїв

Ідентифікатор екрана

ID генерується автоматично при створенні: P1, P2, R1, R12, M1…

Окрім ID кожен екран має два числових параметри:

  • Номер екрана (screenNum) — логічний номер, визначає рядок у таблиці структури. Редагується у властивостях екрана.
  • Пристрій відображення (deviceType) — номер фізичного дисплея (D1, D2…). Встановлюється при створенні, редагується у властивостях.
Навігація: переходи між екранами дозволені лише в рамках одного типу (PC↔PC, R↔R…) і одного пристрою (D1↔D1…).

Структура проекту

Графічний вигляд

Кожен екран відображається як картка з назвою, ID, типом (P/R/M) і номером пристрою. Картки можна вільно перетягувати по канвасу. Декоративні елементи (групи, підписи, стрілки) додаються з лівої палітри drag&drop.

Табличний вигляд

Кнопка ⊞ Таблиця в тулбарі перемикає на матричне представлення:

  • Рядки — номери екранів (screenNum)
  • Стовпці — тип × пристрій (PC D1, PC D2, Render D1…)
  • Клік на комірці — вибирає екран у панелі властивостей
  • Подвійний клік — відкриває редактор елементів

Файлова структура

Дані екранів зберігаються плоско: data/screens/{ID}/screen.json. Структура проекту (список екранів, позиції) — у data/project.json.

Типи елементів

Елементи додаються перетяганням з палітри на полотно. Розмір і позиція задаються у відсотках від розмірів екрана.

Редагування кінцевих точок лінії та стрілки

При виборі елемента типу Лінія або Стрілка з'являються два маркери — ● синій (початок) та ● жовтий (кінець). Перетягніть маркер мишкою для зміни позиції точки всередині bounding box елемента.

Групування елементів

Елемент Група (group_box) охоплює розташовані всередині елементи. При переміщенні групи всі вкладені елементи рухаються разом. Z-index груп нижчий за звичайні елементи, щоб внутрішні елементи залишались інтерактивними.

Параметри (точки)

Кожен параметр системи — це точка з унікальним числовим ID. Зберігається в Redis як хеш point:NNN.

Поля точки

ПолеОпис
idЧисловий ідентифікатор
objectОб'єкт (home, boiler…)
systemПідсистема (heating, gas…)
pointnameНазва параметра (pressure, temperature…)
valueПоточне значення
unitОдиниці виміру (°C, bar, A…)
qualityЯкість сигналу
tsUnix timestamp останнього оновлення

Список усіх точок — Всі параметри. Підтримує фільтрацію за ID, об'єктом, підсистемою, назвою, значенням.

Якість сигналу

Поле quality точки впливає на колір елементів на екрані.

GOOD — норма
WARN — попередження
ALARM — аварія (миготить)
UNCERT — невизначено
INIT — ініціалізація
NODATA — немає даних

Елементи Значення точки, Індикатор, Рівень, Манометр та Підсистема автоматично змінюють колір залежно від якості.

Симулятор

Контейнер infrabox-simulator генерує тестові значення для всіх точок і публікує через MQTT. Дозволяє тестувати без реального обладнання.

Увімкнення / вимкнення

Керується зі сторінки адміністрування: розділ Підсистеми → кнопки Старт / Стоп / Рестарт.

Увага: при увімкненому симуляторі реальні дані можуть перезаписуватись симульованими, якщо обидва джерела публікують одні точки.

Архіватор

Контейнер infrabox-arch безперервно записує значення всіх точок. Доступний за адресою /arch/.

  • Фільтрація за об'єктом, підсистемою, пошук за назвою
  • Вибір кількох параметрів для порівняльних трендів
  • Побудова графіків за довільний проміжок часу

Адміністрування

Сторінка /admin — тільки для адміністраторів. Вкладки:

  • Підсистеми — перегляд стану Docker-контейнерів, старт / стоп / рестарт / ребілд кожної підсистеми
  • Користувачі — перелік облікових записів, зміна паролів та ролей
  • Права доступу — матриця дозволів: які сторінки та об'єкти доступні кожному оператору

Ролі

РольДоступ
adminПовний доступ до всього: перегляд, конструктор, адмін-панель

Контроль доступу

Адміністратор може обмежувати доступ оператора до окремих сторінок та об'єктів даних.

Права на сторінки

У розділі Права доступу адмін-панелі задається список сторінок (наприклад, dashboard), до яких має доступ конкретний оператор. Якщо список не заданий — оператор бачить всі дозволені сторінки.

Фільтрація об'єктів

Поле objects у правах — список об'єктів даних, які оператор може переглядати. API /api/points автоматично фільтрує точки для не-адміністраторів.

JWT-токен містить поточні права. При зміні прав оператору необхідно перезайти, щоб отримати оновлений токен.
